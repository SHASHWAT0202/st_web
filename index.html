Module 01

Introduction, Testing Objectives & Principles

What is Software Testing?

The process of evaluating a software application to identify defects and ensure it meets specified requirements. Software is executed under controlled conditions to detect errors, gaps, or missing functionalities.

Key Definitions (frequently examined):
• Myers: "Testing is executing a program with the intent of finding errors."
• Myers: "A successful test is one that uncovers an as-yet undiscovered error."
• Dijkstra: "Testing can show the presence of bugs but never their absence."
• Cem Kaner: "Software testing is an empirical investigation to provide stakeholders with information about product quality in the context it is intended to operate."

4 Core Testing Objectives

✦ To uncover defects — find bugs at any SDLC stage
✦ Ensure software meets requirements — all features work correctly
✦ Validate & verify system performance — behaves under expected conditions
✦ Improve product quality — reliability, usability, correctness

Typical Objectives (ISTQB):

  • Evaluate work products using static testing (reviews)
  • Verify all specified requirements are fulfilled
  • Validate the system works as stakeholders expect (UAT)
  • Build confidence in quality (highest-risk tests pass)
  • Prevent defects by identifying issues early in requirements

Goals of Software Testing — 3 Categories

Immediate Goals

  • Bug discovery — the more found early, the better

Long-term Goals

  • Quality — correctness, efficiency, reliability
  • Reliability — confidence software won't fail
  • Customer Satisfaction — meets user expectations
  • Risk Management — minimize undesirable events

Post-Implementation Goals

  • Bug Prevention — lessons from past executions
  • Reduced Maintenance Cost — fewer post-release failures
  • Improved Testing Process — knowledge for future projects
Benefits: Early bug detection · Cost reduction · Improved user satisfaction · Increased development efficiency

7 Principles of Software Testing

  1. Testing shows presence of defects, not absence — All tests passing ≠ bug-free. Even 50 passing tests won't catch a crash from a special-character username.
  2. Exhaustive testing is impossible — Infinite input combinations, time, cost constraints. Use risk-based testing, BVA, and equivalence partitioning instead.
  3. Early Testing — Start as early as possible in SDLC. Cost of fixing a defect increases exponentially as the project progresses. V-Model and Agile test in parallel with development.
  4. Defect Clustering — 80% of defects in 20% of modules (Pareto Principle). Focus effort on complex, frequently modified, or newly developed modules.
  5. Pesticide Paradox — Repeating the same test cases won't find new bugs. Regularly review and add new test cases.
  6. Testing is Context Dependent — Healthcare software requires different methods than e-commerce. Domain dictates technique.
  7. Absence of Errors Fallacy — A bug-free system is still useless if it doesn't meet user needs or business requirements.
Exam Tip: All 7 principles are a very common question. Memorise name + one-line explanation.

Coverage Techniques

1. Statement Coverage

Every statement (line) executed at least once.

Coverage = (Executed Statements / Total Statements) × 100

2. Decision/Branch Coverage

Every True/False branch of every decision executed at least once.

Coverage = (Executed Branches / Total Branches) × 100

3. Path Coverage

Every unique execution path from start to end is tested. Most exhaustive.

Coverage = (Executed Paths / Total Possible Paths) × 100
Order of exhaustiveness:
Statement < Branch/Decision < Path
Path = most thorough, most test cases needed
TypeWhat it EnsuresTest Cases
StatementEvery line runs at least onceFew
DecisionBoth outcomes of if/else, loops testedMore than statement
PathAll combinations of branchesMost exhaustive

Validation Techniques

  • UAT (User Acceptance Testing) — end users test in real-world scenarios
  • Usability Testing — evaluates user-friendliness by observing users
  • Performance Testing — speed, responsiveness, stability under load
  • Security Testing — checks vulnerabilities, data protection
  • Compatibility Testing — functionality across different environments/devices
  • Prototyping — working model created for user feedback
  • Simulation — using models to test system behaviour

Software Quality & Reliability

Software Quality
Degree to which software meets specified AND implied needs. Quality ≠ just no bugs — it's "fit for purpose." High quality → user satisfaction, reduced maintenance costs, improved reputation.
Software Reliability
Probability that software performs required functions without failure for a specified time in a specified environment. A reliable product consistently works as expected.
Error → Defect → Failure Chain:
Error (human mistake in code) → Defect (flaw in code) → Failure (observable problem for user)
Example: Spelling mistake in code → button not working → application crash

Common Types of Defects

TypeDescriptionExample
Functional DefectsFeatures don't work as specifiedLogin button doesn't respond
Performance DefectsToo slow or takes too long to loadPage takes 30 seconds to load
Usability DefectsUI confusing, navigation difficultUsers can't find checkout button
Security DefectsVulnerable to unauthorised accessSQL injection possible on login form

Defect Management Process

Step 1Discover — Tester finds & logs defect in Jira/Bugzilla
Step 2Assign & Analyze — Bug assigned to developer who analyses it
Step 3Fix & Test — Developer fixes; sent back to tester for re-verification
Step 4Close — Fix confirmed → bug marked "Closed"

Measuring Quality & Reliability

Reliability Metrics:
• MTTF (Mean Time to Failure) — average time between system failures. Higher = better.
• Defect Density — defects per KLOC (thousands of lines of code). Lower = better.
Defect Density = Defects / KLOC
Example: 30 defects in 12 KLOC = 2.5 defects/KLOC
Quality Metrics:
• Customer Satisfaction — surveys, reviews, feedback
• Post-Release Defects — bugs found after public release

QC vs QA:
QC (Quality Control) = reactive — testing finds & fixes problems
QA (Quality Assurance) = proactive — defines standards to prevent defects

Module 03

Software Testing Life Cycle (STLC)

STLC Overview & Key Points

STLC defines a sequence of activities to ensure software quality. Each phase has defined entry criteria, exit criteria, and deliverables.

Phase 1Requirement Analysis
Phase 2Test Planning
Phase 3Test Case Design
Phase 4Env Setup
Phase 5Test Execution
Phase 6Defect Tracking
Phase 7Test Closure
  • STLC ensures systematic and efficient testing at every SDLC phase
  • Each phase has defined entry and exit criteria for quality control
  • Early QA involvement helps prevent defects, not just detect them

Phase-by-Phase Details

PhaseObjectiveKey ActivitiesDeliverables
1. Req AnalysisUnderstand what to testAnalyse SRS, identify testable items, review with QARTM, Requirement Review Report
2. Test PlanningDefine strategy, scope, objectivesIdentify objectives, schedule, resources, tools, levelsTest Plan Doc, Effort Estimation
3. Test Case DesignCreate detailed test cases & dataWrite TCs for all scenarios, review test scriptsTest Case Doc, Test Data, Automation Scripts
4. Env SetupPrepare testing environmentConfigure servers, databases, tools; smoke testingEnv Setup Confirmation, Smoke Test Results
5. Test ExecutionExecute & record resultsRun TCs, compare actual vs expected, log defectsExecution Report, Defect Reports
6. Defect TrackingTrack defects to closureLog defects, assign severity/priority, verify fixes, regressionUpdated Defect Log
7. Test ClosureSummarise & evaluateAnalyse coverage, defect metrics, lessons learnedTSR (Test Summary Report), Lessons Learned
RTM = Requirement Traceability Matrix — maps requirements to test cases (Phase 1).
TSR = Test Summary Report — final sign-off document (Phase 7).
E-commerce Phase 3: Verify successful order placement with valid card.
Healthcare Phase 3: Verify appointment booking for available doctor slot.
Module 04

Test Case Preparation

What is a Test Case?

A set of conditions or steps to verify that a software feature works as expected. Well-written test cases improve test coverage and repeatability.

8 Components of a Test Case:

  1. Test Case ID
  2. Test Description
  3. Preconditions
  4. Test Steps
  1. Test Data
  2. Expected Result
  3. Actual Result
  4. Status (Pass / Fail)

Positive vs Negative Test Cases

Positive Test Case
Tests with valid data to confirm system works correctly.

Example: Login with correct username and password → Dashboard loads.
Negative Test Case
Tests with invalid data to ensure proper error handling.

Example: Login with wrong password → Error message shown.

Real-Time Examples

TC IDDescriptionPreconditionStepsExpected ResultStatus
TC_EC_01Valid LoginUser registeredEnter valid creds → Click LoginRedirected to dashboard✅ Pass
TC_HC_01Book AppointmentDoctor slot availableSelect doctor → Date/time → ConfirmConfirmation message✅ Pass

Writing Effective Test Cases

Good test cases are:

  • Clear, simple, and unambiguous
  • Traceable to requirements
  • Reusable across builds
  • Independent of others
  • Cover both positive and negative scenarios

Common Mistakes to Avoid:

  • Missing edge cases
  • Writing vague expected results
  • Not updating TCs after requirement changes
  • Ignoring dependencies between modules
Keep TCs updated with every software version change.
Best Practices: Test cases form the foundation of systematic testing · Write clear, traceable, reusable TCs · Always review with peers · Maintain a TC repository for regression. In banking/healthcare, detailed TCs ensure critical operations run safely.

Module 06

Functional vs Non-Functional Testing

Comparison

Functional Testing
Verifies WHAT the system does according to requirements.

Focus: Features and business logic

Examples (online shopping):
• User login works
• Product search works
• Add to cart works
• Payment transaction completes
Non-Functional Testing
Evaluates HOW WELL the system performs.

Focus: Performance, security, usability

Examples (online shopping):
• Website loads within 2 seconds
• System supports 10,000 users
• Secure payment processing
• Reliability under heavy traffic
Key distinction: Functional = WHAT the system does?  |  Non-Functional = HOW WELL does it do it?

Exam Question

A website allows users to place orders successfully, but the page takes 25 seconds to load. Is this a functional issue or non-functional issue?
✅ Non-Functional Issue — The feature works (functional ✓) but performance is poor (non-functional ✗).
Module 07

White Box & Black Box Testing Techniques

White Box Testing

Also known as Clear Box / Structural / Glass Box Testing. Tests internal logic, structure, and code. Performed by developers with programming knowledge.

Objectives:

  • Ensure all independent code paths are executed (Path Coverage)
  • Validate loops, conditions, and data flow
  • Verify internal operations conform to design specifications

Techniques:

TechniqueWhat it Tests
Statement CoverageEvery line of code executed at least once
Branch/Decision CoverageEvery True/False branch executed at least once
Condition CoverageEach condition in a decision tested independently
Path CoverageEvery possible execution path from start to end
Loop TestingZero, one, typical, max, max+1 iterations
Benefits:
• Early defect detection
• Optimised code coverage
• Logical errors easily detected
Limitations:
• Time-consuming
• Requires code-level expertise
• Not suitable alone for large systems

Industrial example — PayPal: Devs ensure all encryption & API calls are triggered, exception handling works, retry loops for failed transactions terminate correctly.

Path Coverage — Code Example

if A > 10:
    if B > 5:
        print("Path 1")   # A>10 AND B>5
    else:
        print("Path 2")   # A>10 AND B≤5
else:
    if B > 0:
        print("Path 3")   # A≤10 AND B>0
    else:
        print("Path 4")   # A≤10 AND B≤0

Independent decisions: A>10 (T/F), B>5 (T/F when A>10), B>0 (T/F when A≤10). Total paths = 4 test cases needed for full path coverage.

Black Box Testing

Also known as Behavioral / Functional Testing. Tests external behaviour without knowledge of internal code. Performed by testers/QA engineers based on requirements.

Objectives:

  • Validate software functionality against requirements
  • Identify incorrect or missing functions
  • Ensure proper user interface behaviour

Techniques:

  • Equivalence Partitioning — divide input into valid/invalid partitions
  • Boundary Value Analysis (BVA) — test at and around boundaries
  • Decision Table Testing — multiple conditions → different actions
  • State Transition Testing — test behaviour when system changes state
  • Use Case Testing — based on user interaction scenarios

Industrial example — Amazon: Verify user can add/remove items correctly, discounts apply properly, payment errors show user-friendly messages, backend failures don't expose sensitive data.

Equivalence Partitioning

Divides input data into partitions. Only one value per partition is tested. Reduces number of test cases significantly.

Example: Age input must be between 18 and 60.

PartitionRangeTest ValueExpected
Valid18–6030Accepted
Invalid (below)< 1817Rejected
Invalid (above)> 6065Rejected

Only 3 test cases needed instead of testing every value.

Tests values at and around the boundaries. Example: Marks between 0 and 100.

Test ValueCategoryExpected

6 test cases for BVA (2 boundaries × 3 values each).

Decision Table Testing & State Transition Testing

Decision Table Testing

A black-box technique used when system behaviour depends on multiple conditions. Represents conditions and actions in tabular form. Ensures all logical combinations are tested.

Structure: Conditions (inputs) + Actions (expected outputs) + Rules (combinations of conditions)

Example: Online Shopping Discount System

Condition / RuleR1R2R3R4
Premium Member?YesYesNoNo
Purchase > ₹5000?YesNoYesNo
20% Discount✓✗✗✗
10% Discount✗✓✓✗
No Discount✗✗✗✓

Rules (Plain English):

  • R1: Premium + Amount > ₹5000 → 20% Discount
  • R2: Premium + Amount ≤ ₹5000 → 10% Discount
  • R3: Non-Premium + Amount > ₹5000 → 10% Discount
  • R4: Non-Premium + Amount ≤ ₹5000 → No Discount

Test Cases:

TCInputExpected Output
TC1 (R1)Premium user + ₹600020% discount
TC2 (R2)Premium user + ₹300010% discount
TC3 (R3)Normal user + ₹700010% discount
TC4 (R4)Normal user + ₹2000No discount
For n conditions → 2ⁿ rules max. 2 conditions → 4 rules. 3 conditions → 8 rules.
Analysis: Are all combinations covered? Which rules correspond to TCs? Can redundant rules be removed?

State Transition Testing

Verifies system behaviour based on states and events. Useful for systems where outputs depend on previous states (ATM, login, traffic lights, vending machines).

ATM PIN Verification Example:

States: Start · Try Again · Access Granted · Account Locked. User gets 3 attempts.
Current StateEventNext State
StartCorrect PINAccess Granted
StartWrong PINTry Again
Try AgainWrong PIN (not 3rd)Try Again
Try AgainWrong PIN (3rd time)Account Locked
Key Test Cases:
• Correct PIN on first attempt
• Incorrect PIN all 3 times → locked
• Correct PIN on 2nd attempt
Analysis Questions:
• Which TCs ensure security?
• Can system move to invalid states?
• Are all transitions tested?
Login System Example: Logged Out → Enter valid creds → Logged In → Click Logout → Logged Out
Module 09

Testing Types: Unit, Integration, UAT, Alpha/Beta, Smoke, Sanity, Regression

Unit Testing

Tests individual components or functions in isolation. Usually by developers using JUnit (Java) or pytest (Python).

Analogy: In a car factory, engineers test the engine, brakes, and headlights separately before assembling. Each part must work independently before integration.
Example: Testing calculate_discount(price, coupon) in isolation before integrating into the checkout module.

Integration Testing

Verifies interaction and data flow between connected modules after integration. Goal: detect interface and communication errors.

Analogy: Chef, waiter, and cashier each work fine alone — but if waiter forgets to tell chef the order, customer stays hungry. That's an integration bug.
Example: Login module correctly passes credentials to Dashboard and retrieves correct user data from the database.

User Acceptance Testing (UAT)

Final phase performed by clients or end users to confirm system meets business requirements and is ready for production.

Feynman: You've built a car — now give it to the customer for a test drive to confirm it feels right.
Example: Bank staff test the new loan application workflow to confirm it matches their actual business process.

Alpha & Beta Testing

Alpha Testing
Internal testing in a controlled environment.
Feynman: You taste the soup yourself before serving it.
Example: QA team tests mobile banking UI in-house.
Finds: Internal defects.
Beta Testing
External testing by real users before official launch.
Feynman: Serve soup to guests, collect feedback before adding to the menu.
Example: Limited user group tests app on their devices, reports usability issues.
Finds: Real-world usability issues.

Smoke Testing

Preliminary test checking build stability. Ensures critical features work before detailed testing begins.

Analogy: Turn on a new machine — if no smoke, safe to test further. Turn on stove before cooking a full meal.
Example: After new deployment, verify login, product search, and checkout work before running full functional tests.
Saves hours by identifying unstable builds early in the testing cycle.

Sanity Testing

Focused testing after minor code fixes or patches. Verifies the specific change works and related features remain unaffected.

Analogy: You repaired car headlights — now check just the lights, not the whole car again. After fixing a door hinge, open/close it to verify — don't recheck the whole house.
Example: After fixing 'coupon not applying' bug, verify discount calculation and checkout process work fine.

Regression Testing

Ensures new changes have not broken existing functionalities in the system.

Analogy: Fixed one loose wire in TV — now recheck volume, color, brightness. One change can affect many areas. Adding a new menu item → chef still tastes old dishes.
Example: After adding wallet payment feature, re-run all critical flows (login, cart, checkout) to confirm stability.

Module 10

Formal, Informal, Monkey, Re-Testing, Load & Stress Testing

Formal Testing

Structured, planned, and documented testing per standards (IEEE 829 / ISO/IEC 29119). Follows entire STLC.

  • Well-defined test plans and strategies
  • Traceability of test cases to requirements
  • Formal reviews and approvals at each stage
  • Standardised templates and reporting formats
Analogy: University final exam — syllabus, exam papers, and grading prepared, reviewed, and approved before use. Ensures standardisation and fairness.
Example: Banking app audit — ISO/IEC standard test plan, each case (e.g., balance validation) reviewed by QA lead, tracked in JIRA, reports signed off by manager and archived for audit.

Informal Testing

Unscripted, undocumented testing without test plans or formal test cases. Provides rapid feedback for early builds/prototypes.

Analogy: Checking car casually before road trip — no checklist, but you spot low fuel or flat tyres. Like proofreading an essay without a formal rubric.
Example: News app prototype — tester opens sections, scrolls, notes missing images/slow loading/layout issues without scripts. Goal: quick feedback before formal testing.

Monkey Testing

Random testing — random inputs and actions to check system stability and crash behaviour. Tests robustness against unpredictable user behaviour.

Analogy: Handing phone to a monkey — it taps everywhere randomly. Pressing random buttons on a TV remote to see if it freezes.
Example: E-commerce app — enter random chars in name/address, click 'Buy Now' repeatedly, rapid page navigation. Reveals input handling and memory management weaknesses.

Re-Testing

Performed after fixing a defect to verify the issue is resolved. Focuses only on previously failed test cases.

Analogy: Fix leaking tap → open tap again to confirm it no longer leaks. Replace flat tyre → test just that tyre for performance.
Example: After resolving login bug, QA re-executes only the failed test case 'Invalid password handling' to confirm fix works.
Re-Testing ≠ Regression Testing. Re-test = verify fix on the specific failed TC. Regression = ensure nothing else broke because of that fix.

Load Testing vs Stress Testing

Load Testing
Measures system performance under expected workload.

Analogy: Testing a bridge under regular traffic.
Analogy: Café runs smoothly with expected customers.

Example: Simulate 1000 concurrent users logging in and browsing.
Stress Testing
Pushes system beyond its limits to see when it breaks.

Analogy: Driving heavy trucks over bridge to find breaking point.
Analogy: Flooding café with sudden crowd to see if service collapses.

Example: Gradually increase to 5000 users to check system limits.

Metrics measured: Response time · Throughput · Server CPU utilisation · Memory usage

Tools: Apache JMeter · LoadRunner

Formal vs Informal:

AspectInformal TestingFormal Testing
PlanningNo planningWell planned
DocumentationMinimal / noneDetailed
ApproachRandom / exploratoryStructured
ReliabilityLess reliableMore reliable
Use caseEarly stages / prototypesFinal validation

Module 11

Ad hoc, Pair, Exploratory, Iterative Testing & Defect Seeding

Ad hoc Testing

Unstructured, informal testing without test cases. Done by experienced testers using intuition, domain knowledge, and creativity. Goal: detect defects quickly outside formal conditions.

Analogy: Trying random controls in a new car — pushing buttons, changing gears, switching modes unexpectedly.
Example: Banking app — rapidly switch screens, input invalid PIN during network lag, perform unexpected swipes and gestures to detect crashes.

Pair Testing

Two testers collaborate on the same system simultaneously. One executes tests, the other observes and documents. Improves defect detection and promotes knowledge sharing.

Feynman: Like two detectives reviewing the same crime scene — one investigates closely, the other evaluates patterns.
Analogy: Two pilots testing aircraft controls — one flying, one monitoring panels. Shared observation increases safety and anomaly detection.
Example: Two testers explore checkout process — one interacts with UI, the other records issues. Used widely in agile teams for fast discovery of edge cases.

Exploratory Testing

Simultaneous learning, designing, and executing tests. Tester explores application freely based on system behaviour. No predefined test cases. Suitable for poorly documented or rapidly changing systems.

Feynman: Like exploring a new city without a map — discovering paths as you go. You learn by trial, error, and curiosity.
Analogy: Unboxing a new smartphone and trying features, menus, settings — figuring out how it works by experimenting.
Example: Tester explores CRM module with incomplete documentation, focuses on discovering workflow issues and hidden bugs.
Best when: Requirements are unclear or rapidly changing. Test design and execution happen at the SAME time.

Iterative Testing

Testing performed repeatedly in cycles after each new build. Ensures fixes don't introduce new errors. Common in Agile, DevOps, and CI pipelines.

Feynman: Like shaping clay — test, adjust, refine, and repeat. Each iteration improves upon previous results.
Analogy: Tasting cake batter after each ingredient and adjusting flavour. A loop of improvement and verification.
Example: Testing app after each sprint update for regression. Repeated tests ensure new features don't break existing ones.

Defect Seeding (Fault Seeding)

Intentionally inserting known defects into the system to measure testing effectiveness and test coverage quality. Used in safety-critical industries (aviation, healthcare).

Feynman: Like a teacher adding hidden mistakes in assignments to see if students catch them. If seeded defects are found, real ones likely will be too.
Analogy: Security team plants fake threats to evaluate emergency readiness — measures how well real issues would be detected.
Example: Healthcare app — team inserts 10 dummy bugs. If 8 are found, test effectiveness ≈ 80%.
Test Effectiveness = (Seeded Defects Found / Total Seeded Defects) × 100

Solved Numericals: 10 defects seeded, 6 found.

  • Effectiveness = 6/10 × 100 = 60%
  • If app has 40 real defects → expected found = 40 × 0.6 = 24 defects
  • If 30 defects found → actual defects in app = 30 / 0.6 = 50 defects

Shortcomings of Defect Seeding:

  • Introduction of unintended bugs into the codebase
  • Artificial defects may distort defect metrics (biggest risk)
  • Time and cost overhead
  • Risk of not removing seeded defects from final product
  • Increased complexity in debugging
  • No direct relation to real defects in the system

Measurements vs Metrics

AspectMeasurementMetric

Analogy: Measurement = "I ran 5 km in 30 mins." (raw data). Metric = "My average speed was 10 km/h." (calculated, useful for comparison)

3 Types of Test Metrics

TypeMeasuresKey Examples
Process MetricsEfficiency of the testing processDRE, Test Coverage, Test Cycle Time
Product MetricsQuality of the product itselfDefect Density, Severity Mix, MTBF
Project MetricsOverall project/test phase healthTC Execution Rate, Defect Arrival Rate, Schedule Variance

1. Process Metrics — Is our way of testing working well?

  • DRE (Defect Removal Efficiency) = (Defects found pre-release) / (Pre-release + Post-release defects) × 100. Higher = better.
  • Test Coverage = (Items tested / Total items) × 100. Higher = better (but 100% ≠ zero bugs).
  • Test Cycle Time = Average time to run a test cycle. Lower = better.

2. Product Metrics — Is the product stable and secure?

  • Defect Density = Defects / Size unit (KLOC). Lower = better.
  • Severity Mix = Fraction of bugs that are critical vs minor. Fewer criticals = better.
  • Reliability (MTBF) = Average time between failures in realistic use. Higher = better.

3. Project Metrics — Are we on track?

  • TC Execution Rate = Tests executed / Time (planned vs actual)
  • Defect Arrival Rate = New bugs per day/week. Healthy project: peaks then declines.
  • Schedule Variance = Actual time − Planned time. Small variance = good.

import unittest
from selenium import webdriver
from selenium.webdriver.common.by import By
import time

class TestEcommerceCheckout(unittest.TestCase):
    def setUp(self):
        self.driver = webdriver.Chrome()
        self.driver.get("http://example-ecommerce.com")
    def tearDown(self):
        self.driver.quit()
    def test_add_item_to_cart(self):
        d = self.driver
        d.find_element(By.ID, "search").send_keys("Laptop")
        d.find_element(By.ID, "add_123").click()
        self.assertIn("Laptop", d.find_element(By.ID, "cart").text)
    def test_checkout_valid_card(self):
        d = self.driver
        d.find_element(By.ID, "add_123").click()
        d.find_element(By.ID, "checkout").click()
        d.find_element(By.ID, "card_number").send_keys("4111111111111111")
        d.find_element(By.ID, "expiry").send_keys("12/25")
        d.find_element(By.ID, "cvv").send_keys("123")
        d.find_element(By.ID, "pay").click()
        time.sleep(2)
        self.assertIn("Payment Success", d.page_source)
